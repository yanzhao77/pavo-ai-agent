# Pavo AI Agent — 技术债务解决方案报告

> 生成日期：2026-07-13 | 基于 Zero-Dependency 改造 v2.0

---

## 一、概述

### 1.1 什么是本次技术债务

改造前，Pavo AI Agent 依赖 **4 个外部基础设施服务**：

```
PostgreSQL + pgvector   →  需要 Docker
Redis                   →  需要 Docker
Celery + Redis Broker   →  需要 Docker
MinIO                   →  需要 Docker
```

这意味着用户必须安装 Docker、拉取 4 个镜像、配置 docker-compose.yml，才能运行这个项目。

**技术债务体现**：用户「开箱即用」的成本太高，外部依赖过多，单人开发场景被过度工程设计。

### 1.2 改造目标

```bash
pip install .
pavo-mcp-server
```

只需两条命令，无需 Docker，无需 PostgreSQL，无需 Redis，无需 Celery，无需 MinIO。

## 二、已偿还的技术债务

### 2.1 数据库层：PostgreSQL → SQLite

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 引擎 | PostgreSQL 16 + pgvector | SQLite + sqlite-vec |
| 驱动 | asyncpg | aiosqlite |
| 向量搜索 | pgvector (IVFFlat索引) | sqlite-vec (vec0虚拟表) |
| 配置 | DATABASE_URL=postgresql://... | 自动 ~/.pavo/pavo.db |
| Docker | 必需 | 不需要 |

**关键变更**：
- `backend/app/db/database.py` 完全重写：sqlite-vec 扩展 + WAL 模式 + vec0 虚拟表
- 新增 `SystemConfig`、`TaskStatus`、`VecMemory` 表
- 提供 `init_db()` 自动初始化

### 2.2 缓存层：Redis → TTLCache

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 实现 | redis-py | cachetools.TTLCache |
| 存储 | 外部 Redis 服务 | 进程内内存 |
| Session TTL | Redis TTL | TTLCache 滑动窗口 |
| 部署 | 需 Docker | 零配置 |

**关键变更**：
- `backend/app/services/cache.py`：PavoCache 封装 TTLCache
- 支持 get/set/delete/touch/size/clear
- 默认 maxsize=10000，ttl=1800s

### 2.3 任务队列：Celery → asyncio Queue

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 框架 | Celery 5.4 | asyncio.Queue |
| Broker | Redis | 无需 |
| Worker | celery worker 进程 | 后台 asyncio Task |
| 并发控制 | --concurrency=N | Semaphore (默认3) |
| 状态持久化 | Redis result backend | SQLite task_status 表 |

**关键变更**：
- `backend/app/services/celery_app.py` 删除
- `backend/app/services/task_queue.py` 新增
- `backend/app/services/video_tasks.py` 从 `@shared_task` 改为 `async def`

### 2.4 文件存储：MinIO → 本地文件

| 维度 | 改造前 | 改造后 |
|--------|--------|--------|
| 实现 | boto3 + MinIO | LocalStorageClient |
| 存储位置 | MinIO Bucket | ~/.pavo/storage/ |
| 公开访问 | MinIO URL | FastAPI StaticFiles |
| 部署 | 需 Docker | 零配置 |

**关键变更**：
- `backend/app/services/local_storage.py` 合并到 `storage.py`
- 删除 boto3/MinIO 依赖
- 支持 upload/delete/get_url/mount_static

### 2.5 依赖清理

| 操作 | 依赖 |
|------|------|
| 删除 | asyncpg, psycopg2, redis-py, celery, boto3, minio |
| 新增 | aiosqlite, sqlite-vec, cachetools, click |
| 保留 | fastapi, uvicorn, sqlalchemy, alembic, openai, httpx |

**效果**：requirements.txt 从 12 行缩减到 11 行，移除了所有基础设施依赖。

## 三、剩余技术债务

### 3.1 P1 - 高优先级

| 编号 | 债务项 | 说明 | 影响范围 | 建议方案 |
|------|--------|------|---------|---------|
| TD-01 | sqlite-vec 在部分环境不可用 | 需要 libsqlite3 支持扩展加载 | 向量搜索功能 | 增加启动检查 + 降级到基于 JSON 的向量搜索 |
| TD-02 | 缺少数据库迁移工具 | SQLite 模式下 Alembic 迁移未适配 | 数据库版本管理 | 增加 SQLite 迁移脚本或使用 schema version 表 |
| TD-03 | 任务队列状态无自动清理 | task_status 表持续增长 | 长期运行稳定性 | 增加定时清理 `WHERE created_at < now() - 7d` |

### 3.2 P2 - 中优先级

| 编号 | 债务项 | 说明 | 影响范围 | 建议方案 |
|------|--------|------|---------|---------|
| TD-04 | 缺乏跨进程缓存同步 | 多进程部署时 TTLCache 各自独立 | 多 Worker 场景 | 增加可选的 Redis 回退 or 文件缓存 |
| TD-05 | SQLite 并发写入锁 | WAL 模式但仍受限于单文件写入 | 高并发视频生成 | 增加写入队列 or 连接池重试 |
| TD-06 | 启动脚本未检查 Node.js 版本 | pavo-start 未验证前端依赖 | 前端启动失败 | 启动时执行 `node -v` && `npm -v` 检查 |
| TD-07 | MCP Tool 名称未按规范统一 | 部分工具使用 `pavo_` 而非 `pavo_tool_` 前缀 | MCP 协议兼容性 | 全局重命名工具名称 |

### 3.3 P3 - 低优先级

| 编号 | 债务项 | 说明 | 建议方案 |
|------|--------|------|---------|
| TD-08 | 测试覆盖率不足 (Edge cases) | 向量搜索失败降级、Cache 满清理等 | 补全边缘测试用例 |
| TD-09 | 日志输出未统一格式 | 部分模块用 print，部分用 logger | 统一日志格式 [pavo] [module] message |
| TD-10 | 文档中 Docker 依赖未完全清理 | README 仍有 docker-compose 引用 | 更新 README 启动方式 |

## 五、改造前后对比

### 5.1 用户启动成本

```
改造前：
  1. 安装 Docker Desktop
  2. docker compose pull (4 个镜像, ~2GB)
  3. docker compose up -d
  4. 创建虚拟环境
  5. pip install -r requirements.txt
  6. uvicorn app.main:app

改造后：
  1. pip install .
  2. pavo-start
```

### 5.2 依赖对比

| 指标 | 改造前 | 改造后 |
|------|--------|--------|
| 外部服务依赖 | 4 (PG/Redis/Celery/MinIO) | 0 |
| requirements.txt 行数 | 12 | 11 |
| Docker 依赖 | 必需 | 可选 |
| 首次启动时间 | ~5 分钟 (含镜像拉取) | ~10 秒 |
| 磁盘占用 | ~2GB (Docker 镜像) | ~50MB |

## 六、总结

本次零依赖改造偿还了核心技术债务：**将 Pavo AI Agent 从 4 个外部服务的依赖中解放出来**。用户现在可以通过 `pip install . && pavo-mcp-server` 两条命令直接运行。

剩余 10 项债务（3 项 P1、4 项 P2、3 项 P3）已制定修复计划，预计 8 天 Sprint 可全部解决。
