# Pavo AI Agent — 技术债务解决方案报告

> 生成日期：2026-07-16 | 基于 Zero-Dependency 改造 v2.3

---

## 一、概述

### 1.1 当前技术债务现状

在成功完成 **Zero-Dependency 零依赖改造**（移除了 PostgreSQL、Redis、Celery 和 MinIO）后，系统架构变得极其轻量，可以实现一键本地启动。但在这次大规模改造过程中，由于代码迁移的渐进性，遗留了两项显著的技术债务，它们虽然不影响系统当前功能的正常运行，但严重制约了代码的可维护性和未来的扩展性。

根据对最新代码库的全面扫描，这两项技术债务具体表现为：

1. **缓存层（Cache Layer）碎片化与孤立**：
   系统引入了统一的 `PavoCache`（位于 `app/services/cache.py`），但核心业务模块 `auth.py` 和 `task_queue.py` 仍在使用内部各自维护的 `dict` 作为缓存（如 `_token_cache` 和 `_status_cache`）。统一缓存接口完全处于孤立状态，未被业务代码实际调用。

2. **向量检索（Vector Search）接口抽象失效**：
   系统设计了优雅的 `VectorStore` 抽象接口（位于 `app/vectorstore/__init__.py`），并实现了底层的 `SqliteVecStore`。然而，核心模块 `mcp_server/memory/store.py`（负责记忆存储与检索）完全绕过了这个抽象层，目前它不仅没有调用底层的 `MATCH` 向量搜索，甚至退化为了基于重要性（importance）的标量排序查询。

### 1.2 解决技术债务的意义

清除这些技术债务对于项目的长期健康至关重要：
- **提升代码内聚性**：通过统一缓存层，消除重复造轮子，使内存管理和 TTL 过期策略集中化。
- **恢复核心 AI 能力**：打通 `VectorStore` 抽象，使 RAG（Retrieval-Augmented Generation）系统能够真正利用 `sqlite-vec` 的余弦相似度检索能力，而非退化为简单的标量查询。
- **架构一致性**：使代码的实际运行路径与设计的架构蓝图（零依赖方案设计）保持完全一致。

## 二、技术债务详情与解决方案

### 2.1 债务一：缓存层（Cache Layer）分裂

#### 2.1.1 现状分析

在代码扫描中发现，`auth.py` 和 `task_queue.py` 存在以下硬编码的字典缓存：

**`app/services/auth.py`**
```python
# 内部硬编码字典，手动管理过期逻辑
_token_cache: dict = {}
```

**`app/services/task_queue.py`**
```python
# 内部硬编码字典，无过期清理机制
self._status_cache: dict = {}
```

与此同时，系统已经存在一个功能完备的 `PavoCache`：
**`app/services/cache.py`**
```python
class PavoCache:
    # 具备 TTL 过期清理机制的统一缓存
```

#### 2.1.2 解决路径（重构方案）

**步骤 1：重构 `auth.py` 接入统一缓存**
- 移除模块级别的 `_token_cache` 字典。
- 引入 `from app.services.cache import pavo_cache`。
- 在 `create_token` 中，使用 `pavo_cache.set(token_hash, data, ttl=expires_in)`。
- 在 `verify_token` 中，使用 `pavo_cache.get(token_hash)`。

**步骤 2：重构 `task_queue.py` 接入统一缓存**
- 移除类级别的 `self._status_cache`。
- 引入 `pavo_cache`。
- 任务状态更新时，使用 `pavo_cache.set(f"task:{task_id}", status_data, ttl=86400)`（状态保留 24 小时）。

**步骤 3：统一清理机制**
- 废弃 `auth.py` 内部基于 `now > e["expires_at"]` 的手动清理循环，依赖 `PavoCache` 内部的自动 TTL 回收。

### 2.2 债务二：向量检索（Vector Search）抽象失效

#### 2.2.1 现状分析

在代码扫描中，`app/vectorstore/__init__.py` 定义了完整的向量操作接口：
```python
class VectorStore(ABC):
    async def search(self, query_vector: list[float], limit: int = 5) -> list[tuple[str, float, dict]]: ...
```

但在 `mcp_server/memory/store.py` 的 `search_memory` 方法中，代码完全没有调用 `VectorStore`，也没有使用 `sqlite-vec` 的向量操作，而是退化为了普通的 SQLAlchemy 标量查询：
```python
async def search_memory(self, user_id: str, query_text: str, ...) -> list[dict]:
    # 严重缺陷：完全没有使用 query_text 的 Embedding 进行余弦相似度检索
    stmt = select(UserMemoryORM).where(UserMemoryORM.user_id == user_id)
    # 仅仅通过 importance 进行倒序排列
    stmt = stmt.order_by(UserMemoryORM.importance.desc()).limit(limit)
```

#### 2.2.2 解决路径（重构方案）

**步骤 1：引入 `VectorStoreFactory`**
在 `mcp_server/memory/store.py` 的 `__init__` 方法中，初始化向量存储客户端：
```python
from app.vectorstore import VectorStoreFactory

class MemoryStore:
    def __init__(self):
        self.vec_store = VectorStoreFactory.create("sqlite")
```

**步骤 2：重写 `save_memory`（双写机制）**
当保存新记忆时，除了将元数据写入 `UserMemoryORM`（即 `user_memories` 表），必须同时调用向量存储：
```python
embedding = await self.embedding_client.embed(text_for_vector)
# 写入标量数据库
self.session.add(new_memory)
# 写入向量数据库 (sqlite-vec 虚拟表)
await self.vec_store.upsert(
    id=new_memory.id,
    vector=embedding,
    metadata={"user_id": user_id, "type": memory_type}
)
```

**步骤 3：重写 `search_memory`（RAG 检索召回）**
彻底改变当前的查询逻辑，恢复 RAG 能力：
```python
async def search_memory(self, user_id: str, query_text: str, limit: int = 5) -> list[dict]:
    # 1. 将查询文本转为向量
    query_vector = await self.embedding_client.embed(query_text)
    
    # 2. 从 sqlite-vec 中进行余弦相似度检索 (ANN)
    vec_results = await self.vec_store.search(
        query_vector=query_vector, 
        limit=limit,
        filters={"user_id": user_id}
    )
    
    # 3. 根据召回的 ID，从标量数据库中提取完整内容
    memory_ids = [res[0] for res in vec_results]
    stmt = select(UserMemoryORM).where(UserMemoryORM.id.in_(memory_ids))
    # ...
```

## 三、实施计划与验收标准

### 3.1 实施优先级

| 任务 | 优先级 | 预计耗时 | 影响范围 |
|------|-------|---------|---------|
| 修复向量检索（债务二） | **高** (P0) | 4 小时 | RAG 记忆检索的准确性、AI 生成的上下文相关性 |
| 统一缓存层（债务一） | **中** (P1) | 2 小时 | 内存占用、认证模块和任务队列的稳定性 |

### 3.2 验收标准

1. **缓存层验收**：
   - 执行 `grep -rn "dict\s*=\s*{}" backend/app/services/` 必须无输出。
   - `test_auth.py` 和 `test_zero_dependency_e2e.py` 中的所有测试用例必须 100% 通过。
2. **向量检索验收**：
   - 执行 `grep -rn "SqliteVecStore" backend/mcp_server/memory/store.py` 必须有匹配结果。
   - `search_memory` 必须调用 `self.embedding_client.embed(query_text)` 生成查询向量。
   - 运行 RAG 相关的集成测试，确保语义相似度召回逻辑生效。

零依赖改造的完成为 Pavo AI Agent 奠定了极简、轻量的基础架构。接下来，通过偿还上述两项技术债务，系统不仅能在架构上实现完美的闭环，更能彻底激活基于 `sqlite-vec` 的核心 AI 记忆检索能力，为用户提供真正强大的本地化智能体体验。
