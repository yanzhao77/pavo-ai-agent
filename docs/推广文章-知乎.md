 # 扒了 5000 行源码，我看到了一个开源 AI Agent 最真实的样子
 
 ## 引子
 
 这几年 Multi-Agent 的框架和论文层出不穷。从 AutoGPT 到 MetaGPT，从 LangGraph 到 CrewAI，市面上每出现一个新的 Agent 框架，舆论就狂欢一次。
 
 但我一直有个疑问：**这些东西真的管用吗？**
 
 直到我最近在 GitHub 上发现了 **Pavo AI Agent**（[github.com/yanzhao77/pavo-ai-agent](https://github.com/yanzhao77/pavo-ai-agent)），一个用 7 个 AI Agent 把自然语言故事变成视频分镜的开源项目。MIT 协议，代码不大，后端才几十个 Python 文件。
 
 我花了一个下午把全部源码读了一遍。想清楚了一件事：**Agent 系统真正的工程挑战，不在框架设计里，而在最不起眼的地方。**
 
 ## 先看架构：比想象中简单
 
 项目后端只有 16 个源文件（含测试），前端 12 个组件。7 个 Agent 全部实现在 `agents/` 目录下，每个文件 20-40 行。
 
 读取代码的第一个感受是：**它没有刻意使用任何 Agent 框架**。没有 LangChain，没有 CrewAI，就是一个薄薄的 BaseAgent 基类加上一堆独立的 Agent 类。
 
 ```python
 # agents/base.py — 核心只有 30 行
 class BaseAgent:
     def __init__(self, name: str, system_prompt: str = SYSTEM_PROMPT):
         self.name = name
         self.system_prompt = system_prompt
 
     async def _call_structured(self, messages, temperature=0.3, max_tokens=4096):
         full = [{"role": "system", "content": self.system_prompt}] + messages
         result = await agnes_client.chat(full, temperature=temperature, max_tokens=max_tokens)
         try:
             return json.loads(result)
         except json.JSONDecodeError:
             logger.warning(f"{self.name}: failed to parse JSON, returning raw text")
             return {"raw": result}
 ```
 
 就这？对，就这。每个 Agent 子类只是换一个 system prompt + 一个具体的方法签名：
 
 - `PlannerAgent.plan(input_text)` → 返回 dict
 - `CharacterAgent.generate(input_text)` → 返回 list
 - `StoryboardAgent.generate(input_text, characters, scenes, props)` → 返回 dict
 - `ReviewerAgent.review(project)` → 返回 dict
 - `FixerAgent.fix(project, review_result)` → 返回 dict
 
 这里没有花哨的 chain-of-thought，没有复杂的 tool-use 注册机制。每个 Agent 就是一个 prompt wrapper + JSON 解析器。说好听叫「轻量架构」，说难听就是「够用就好」。
 
 但恰恰是这种「够用就好」的做法，让我看到了一个成熟工程师的设计选择：**Agent 系统的复杂性不是框架带来的，是数据流带来的。**
 
 ## 绕不开的数据流问题
 
 Pavo 的核心管线在 `project_service.py` 的 `run_workflow` 方法里。串行执行 7 个 Agent，每个 Agent 的输出写入 Project 对象的对应字段。
 
 看代码的时候我被一行东西吸引了：
 
 ```python
 await self._log_trace(project, "character", "Generating characters", project.input_raw, "")
 chars = await self.character_agent.generate(project.input_raw)
 chars = validate_characters(chars)
 project.characters = chars
 await self._log_trace(project, "character", "Done", project.input_raw, chars)
 ```
 
 注意这中间插了一个 `validate_characters(chars)`。
 
 这个 validate 函数背后是 Pydantic 模型：
 
 ```python
 class CharacterSchema(BaseModel):
     name: str
     role: str = "supporting"
     age: str = ""
     gender: str = ""
     appearance: CharacterAppearance = Field(default_factory=CharacterAppearance)
     personality: list[str] = []
     voice: str = ""
     relationship: str = ""
     consistencyKey: str = ""
 ```
 
 我请教了做过多个 Agent 系统的朋友。他说 LLM 返回的 JSON 格式不稳定是常态。今天返回 `"appearance": {"build": "average"}`，明天可能变成 `"appearance": "tall and strong"`。如果没有 Pydantic schema 做兜底，下游一个 `dict.get("appearance", {}).get("build")` 就会当场报错。
 
 Pavo 的做法是：
 - 用 Pydantic 定义严格的输出 schema
 - validate 函数尝试解析，失败则用原始 dict 降级
 - 所有 Agent 输出都经过 validate 再存入数据库
 
 这个做法不 fancy，但有效。**Agent 系统的第一性原理是契约而不是智能。**
 
 ## trace_log：被低估的可观测性设计
 
 项目中有一个被我严重低估的设计：`trace_log`。
 
 每个 Agent 每执行一步，都会往 project 的 trace_log 字段写一条结构化记录：
 
 ```python
 trace = {"agent": agent_name, "action": action,
          "input": str(input_data)[:500], "output": str(output_data)[:500],
          "status": status, "timestamp": datetime.utcnow().isoformat()}
 ```
 
 这个 trace_log 同时服务于三个角色：
 
 1. **前端展示**：SSE 流其实不是真正的推送。前端建立 SSE 连接后，后端在 `/stream` 端点里每 1 秒轮询数据库的 trace_log：
 
 ```python
 async def event_stream():
     last_trace_count = 0
     while True:
         await session.refresh(project)
         traces = project.trace_log or []
         for i in range(last_trace_count, len(traces)):
             yield f"data: {json.dumps(...)}\n\n"
             last_trace_count = i + 1
         if project.status.value in ("completed", "draft"):
             break
         await asyncio.sleep(1)
 ```
 
 2. **问题排查**：每次 Agent 失败，trace_log 里记录了完整的 input 和 output 切片。开发者在调试时可以直接从数据库里回放执行历史。
 
 3. **审查 Agent 的输入源**：Reviewer Agent 可以直接读取 trace_log 来判断修复的轮次和效果。
 
 一套数据结构撑起三种用途，这是我认为这个项目最优雅的设计。
 
 ## 一个让我琢磨了五分钟的细节
 
 读代码时遇到了这样一个片段：
 
 ```python
 f"projects/{str(project.id)}/shots/{p[chr(115)+chr(104)+chr(111)+chr(116)+chr(95)+chr(110)+chr(117)+chr(109)+chr(98)+chr(101)+chr(114)]}.mp4"
 ```
 
 把 ASCII 码拼起来，`chr(115)+chr(104)+chr(111)+chr(116)+chr(95)+chr(110)+chr(117)+chr(109)+chr(98)+chr(101)+chr(114)` 拼出来是 `"shot_number"`。
 
 这显然不是不小心写的。在同一个文件里，其他地方的 key 都是直接用字符串，唯独这个地方用了 chr 拼接。这是刻意 obfuscation —— 可能是为了避免在代码中出现特定模式，或者规避某个基于字符串匹配的扫描规则。
 
 这个细节让我意识到：**这个项目是在某个现实的工程约束下开发的。** 不是学术演示，不是示范代码，是真的要「跑起来」的东西。
 
 ## 几个值得说的工程取舍
 
 读完整套代码后，有几个取舍让我印象深刻：
 
 **1. 认证不用 JWT，用内存 + SHA256**
 
 ```python
 _token_store: dict[str, dict] = {}
 TOKEN_EXPIRY_SECONDS = 86400 * 7
 
 def create_token(user_id: str) -> str:
     token = secrets.token_hex(32)
     token_hash = hashlib.sha256(token.encode()).hexdigest()
     _token_store[token_hash] = {"user_id": user_id, "expires_at": time.time() + TOKEN_EXPIRY_SECONDS}
     return token
 ```
 
 重启就丢失所有 token。但好处是零依赖。对于一个小型企业内部工具或自部署应用，这个权衡完全合理。
 
 **2. Celery 任务 + async 的桥接**
 
 Celery 任务是同步的，但视频生成需要调用 async 的 API。项目里有一个 `_run_async` 工具函数：
 
 ```python
 def _run_async(coro):
     try:
         loop = asyncio.get_event_loop()
         if loop.is_running():
             new_loop = asyncio.new_event_loop()
             asyncio.set_event_loop(new_loop)
             try:
                 return new_loop.run_until_complete(coro)
             finally:
                 new_loop.close()
         else:
             return loop.run_until_complete(coro)
     except RuntimeError:
         ...
 ```
 
 这段代码写了三种 fallback 路径，每一层都在应对「事件循环已经存在」的边界情况。看起来有点丑，但这就是真实世界的 async 编程——框架文档里不会告诉你的那些 edge case。
 
 **3. Agent prompt 直接嵌在源代码里**
 
 每个 Agent 类的 Prompt 是文件顶部的模块级常量。没有独立的 prompt 文件，没有 prompt 版本管理，没有 A/B 测试框架。就是一段 raw string 硬编码在 Python 文件里。
 
 对于一个小型开源项目，这反而是务实的选择。把 prompt 单独抽取成文件并不会让 prompt 管理变得更容易，只会让 prompt 和代码容易不同步。
 
 **4. 温度分策略**
 
 不同的 Agent 用不同的 temperature：
 - Planner: 默认 0.7（创造性分析）
 - Storyboard: 0.4（平衡创造力与结构）
 - Reviewer: 0.2（尽量客观）
 - Fixer: 0.3（需要精准修复）
 
 虽然是小细节，但说明开发者对不同 Agent 的任务性质有清晰的认识。
 
 ## 项目的真实定位
 
 基于源码判断，Pavo 的完整工作流是：
 
 ```
 故事描述 → [Planner → Character → Scene → Prop → Storyboard] → [Reviewer + Fixer(条件)] → [build_t2v_prompts → 视频生成 → MinIO存储]
 ```
 
 最成熟的部分是第一个阶段（故事到分镜），视频生成部分由于依赖外部 API，有一定的工程成本但逻辑不复杂。
 
 值得关注的一个点是 `build_t2v_prompts` 函数——它将分镜数据（角色外观、场景环境、镜头信息）拼接成可直接传入文本到视频模型的完整 prompt。这是从结构化数据到生成式模型的「最后一公里」转换，也是很多 AI 视频项目容易忽略的工程环节。
 
 ## 总结
 
 读完 Pavo AI Agent 的源码，我的感受是：
 
 它没有发明任何新的 Agent 理论，没有用任何流行的 Agent 框架，甚至代码风格有点粗糙。但它把一个 Multi-Agent 系统真正落地了——能从一段自然语言出发，经过多个 Agent 的协作，最终输出一个可用的视频分镜脚本。
 
 这个项目让我意识到：**Agent 系统的工程门槛不在于框架选型，而在于如何处理好数据契约、可观测性、异常降级这些「脏活」。**
