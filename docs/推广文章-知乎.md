## 一、先聊 MCP：让 AI 工具能"操作"Pavo

如果你看过我之前的文章，应该记得 Pavo 的传统用法是打开浏览器操作。这对于人来说没问题，但对于 AI 编程工具（Cursor、Claude Code）来说，等于"此路不通"。

v2.3 的 MCP Server 解决了这个问题。

### 它做了什么

简单说，MCP Server 是一个"中间人"进程。它把 Pavo 的所有能力注册成 12 个工具，AI 编程工具可以通过 MCP 协议直接调用这些工具。

这意味着什么？

你在 Cursor 里写代码时，可以直接说：

> "用 Pavo 创建一个关于父子亲情的视频分镜项目"

Cursor 会自动调用 `pavo_create_project` 工具 → 启动 7 Agent 管线 → 返回分镜数据。全程不需要打开浏览器。

### 代码有多简单

整个 MCP Server 的核心入口只有 200 行：

```python
@server.list_tools()
async def list_tools():
    return [
        {"name": "pavo_create_project", ...},
        {"name": "pavo_get_project", ...},
        # ... 共 12 个
    ]

@server.call_tool()
async def call_tool(name, arguments):
    result = await dispatch(name, arguments)
    return result
```

MCP SDK 会处理 stdio/SSE 通信、JSON 序列化、错误码映射等基础设施。你需要做的只是定义工具和处理逻辑。

### 我在意的一个设计细节

MCP Server 的返回值采用统一的 `MCPToolResult` 格式：

```python
class MCPToolResult(BaseModel):
    success: bool
    data: dict | list | None = None
    error: MCPError | None = None
```

这看起来平淡无奇，但实际开发中我发现它**大幅降低了调试成本**。所有工具返回结构一致，错误码统一（NOT_FOUND / INVALID_PARAM / INTERNAL_ERROR...），AI Client 不需要为每个工具写不同的错误处理逻辑。

## 二、再聊 Memory + RAG：Agent 终于有了"记忆"

这可能是 v2.3 最值得关注的部分。

### 解决了什么问题

用 Pavo 创作分镜时，之前的流程是：

1. 用户输入故事 → 7 Agent 工作 → 输出分镜
2. 用户换一个故事 → 7 Agent 从头工作 → 输出分镜

**每次都是"从零开始"**。用户之前调整的偏好、常用的角色风格、偏爱的场景类型，全部丢失。

这就像你每次去同一家理发店，理发师都要问"你想要什么发型"——即使你上周刚剪过。

### 双层记忆架构

v2.3 实现了两层记忆：

**短期记忆**（Redis，TTL 30 分钟滑动窗口）：
- 保存当前会话的对话历史
- 每次 Tool 调用自动刷新 TTL
- 会话结束后自动提取重要信息

**长期记忆**（PostgreSQL + pgvector，永久）：
- 6 种记忆类型：style / character / scene / preference / story_arc / feedback
- 记忆内容向量化存储（1536 维）
- 重要性评分机制

### 重要性评分：不是所有记忆都值得保留

这是我觉得最有意思的设计。代码在 `memory/importance.py` 里，只有 30 行：

```python
class ImportanceStrategy:
    @staticmethod
    def from_user_saved():
        return 0.9  # 用户主动保存的高价值

    @staticmethod
    def from_auto_extracted(mentions, recency_days):
        # 出现 3 次的内容比出现 1 次的重要
        # 昨天产生的内容比一个月前的重要
        return min(0.3 + mentions * 0.1 - recency_days * 0.01, 0.8)

    @staticmethod
    def from_feedback_derived(rating):
        return 0.7 if rating == "up" else 0.2
```

为什么重要？因为如果没有这个机制，记忆系统存储的垃圾会远超有价值的信息。**importance < 0.2 且超过 30 天的记忆会自动清理。**

### RAG 知识库：60 条影视专业知识

7 个 Agent 生成分镜时，普通用户对"景别"、"运镜"的认知是模糊的。RAG 知识库就是为解决这个问题而生——在 Storyboard Agent 生成分镜时，自动检索并注入相关知识：

```
用户输入 → Embedding → pgvector 检索 → Re-ranker → 注入 Prompt
```

效果：分镜中专业术语的正确率显著提升，镜头语言的规范性有了保障。

## 三、最后说 Workflow 可视化：看得见的管线

之前的 Agent 执行状态只有文本日志：

```
[Planner] Analyzing... ✓
[Character] Generating... ✓
[Scene] Generating... ✓
```

v2.3 改成了 SVG 管线图：

```
                 ┌──────────┐
   用户故事 ────►│  Planner  │──► 0.8s ✓
                 └──────────┘
                       │
                       ▼
                 ┌──────────┐
                 │Character │──► 1.2s ✓
                 └──────────┘
              ... (共 7 个节点)
```

每个节点的颜色代表状态（灰/蓝/绿/红），点击节点可以看到详细信息（输入摘要/输出摘要/耗时/错误信息）。底部还有一个时间线条，直观展示每个 Agent 的耗时比例。

技术实现上选择了**纯 SVG + CSS 动画**，没有引入 React Flow 或 ECharts。原因是管线拓扑固定（线性 7 节点），不需要动态布局和拖拽编辑，纯 SVG 方案更轻量。

## 四、一些观察和思考

### 架构演进的方向

```
v1.0: Web 应用             —— 人能用了
v2.0: MCP 接口              —— AI 工具能用了
v2.3: Memory + RAG + 可视化 —— 用得好了
```

这个演进路径值得借鉴：**先让核心功能跑通，再标准化接口，最后做智能增强。**

### 一点工程上的体会

重写 Pavo 的文章时我注意到一个细节：整个 v2.3 的 Memory 模块，核心代码不到 500 行。MCP Server 入口不到 200 行，Memory Middleware 不到 100 行。

**很多看似复杂的能力，用对了方法，实现起来并不复杂。**

关键在于清晰的架构分层：

```
表示层 (MCP Tools) → 中间层 (Middleware) → 存储层 (Memory Store) → 基础设施 (pgvector)
```

每一层只做一件事，接口清晰，替换起来也容易。比如今天用 pgvector，明天想换 Qdrant，只需要改存储层。

### 下一步

v3.0 已经在规划中，主要方向是视频 Agent —— 从"生成分镜"到"智能创作视频"，包括 Prompt 优化、多版本管理、质量检测。
